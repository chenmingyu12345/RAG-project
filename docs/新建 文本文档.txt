第一部分：RAG核心原理


1.1 什么是RAG？

RAG（检索增强生成）让大模型在回答问题时，先从知识库检索相关信息，再基于这些信息生成答案。核心价值：把大模型从“闭卷考试”变成“开卷考试”。

1.2 RAG五环节架构

┌─────────────────────────────────────────────────────────────────────┐
│                    离线预处理（用户提问前执行）                      │
│  原始文档 → 【分片】→ 【索引】→ 向量知识库                           │
└─────────────────────────────────────────────────────────────────────┘
                                    ↓
┌─────────────────────────────────────────────────────────────────────┐
│                    在线问答（用户提问后执行）                        │
│  用户提问 → 【召回】→ 【重排】→ 【生成】→ 答案                      │
└─────────────────────────────────────────────────────────────────────┘

第四部分：提示词清单 + 解析


提示词①：分片模块

提示词：

请实现文档分片模块，负责加载 docs 文件夹下的所有文档，将其切割为适合检索的文本片段。要求：

1. 支持 .txt 和 .pdf 格式
2. 使用 RecursiveCharacterTextSplitter，参数：
   - chunk_size=500
   - chunk_overlap=50
   - separators=["\n\n", "\n", "。", "！", "？", "；", "，", " ", ""]
3. 打印分片统计信息
4. 返回分片后的文档列表
5. 输出文件：backend/app/core/document_loader.py

设计解析：

设计点
解析
chunk_size=500
每片500字符。太大检索不精准，太小丢上下文，500是经验值
chunk_overlap=50
片间重叠50字符。防止关键信息被切断在分片边界
separators 层级
段落→句子→标点→词，按优先级切分，最大限度保语义完整
指定输出路径
让TRAE把代码生成在指定位置，保持项目结构清晰

对应RAG环节：【分片】——将完整长文档切割为多个独立文本片段，为后续索引和检索提供基础单元。


提示词②：索引模块

提示词：

请实现向量索引模块，负责将分片后的文本通过 Embedding 模型转化为向量，并存入向量数据库。要求：

1. 使用 LangChain 的 OpenAIEmbeddings（兼容 DeepSeek 接口）
2. 实现创建向量库功能：遍历分片 → Embedding转为向量 → 存入FAISS
3. 实现加载向量库功能：从 knowledge_base 目录加载
4. 实现相似度检索功能：接收问题和k值，返回Top-K片段
5. 从 .env 读取 DEEPSEEK_API_KEY 和 OPENAI_BASE_URL
6. 输出文件：backend/app/core/vector_store.py

设计解析：

设计点
解析
OpenAIEmbeddings
用兼容OpenAI接口的Embedding模型，DeepSeek完全兼容
遍历→向量化→存入FAISS
三步完整流程：逐片处理、文本转数字向量、成对存储
加载已有向量库
避免每次启动都重新索引，提升效率
相似度检索
提供检索接口供后续问答环节调用，k值可配置
从.env读取
API Key不写死在代码中，防止泄露

对应RAG环节：【索引】——通过Embedding将文本语义数字化，构建可快速检索的向量知识库。


提示词③：RAG引擎

提示词：

请实现 RAG 问答引擎，串联召回、重排、生成三个环节。要求：

1. 召回：从向量库检索 Top10 相关片段，使用余弦相似度
2. 重排：从 Top10 中精选 Top3 最优片段（使用 LLM 辅助打分）
3. 生成：将「用户问题+Top3片段」组合为Prompt，调用 DeepSeek-v4-pro 生成答案
4. 返回：{answer, sources, retrieved, reranked}
5. 打印每个环节的执行信息
6. 输出文件：backend/app/core/rag_engine.py

设计解析：

设计点
解析
召回Top10
向量相似度粗筛。快、成本低，从全量文档快速捞取候选
重排选Top3
高精度精选。慢、成本高，从10条中过滤噪音
余弦相似度
关注向量“方向”而非“长度”，更适合文本语义匹配
LLM辅助打分
生产环境用CrossEncoder，这里用LLM打分降低实现门槛
deepseek-v4-pro
DeepSeek旗舰模型，回答质量最高
全链路日志
展示召回→重排→生成每一步的中间结果，便于调试
返回sources
展示参考来源，体现RAG的“可溯源”特性

对应RAG环节：【召回】+【重排】+【生成】——用户提问后，先粗筛（快），再精筛（准），最后大模型基于精选素材生成答案。


提示词④：Webhook服务

请实现 Web 服务模块，提供 RESTful API 接口。要求：

5.2 为什么召回+重排两步不能少？

环节
作用
如果跳过会怎样
召回（粗筛）
快速捞取候选
全量文档直接精排→计算成本极高、响应极慢
重排（精筛）
精准过滤噪音
只有召回→答案准确率低，噪音多

核心设计逻辑：召回是“快”，重排是“准”，两者配合才能“又快又准”。
